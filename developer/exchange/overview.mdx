The hosted bunnyOS Exchange API is a REST service for everything *around* a bunny's market: authentication, market and account reads, profile metadata, and the social layer (threads, comments, votes, moderation). It is the read-and-write surface for application data, it is **not** where trades settle.

Trading bunny keys (buying and selling) happens **on-chain against the bonding-curve contract on Base**, not through this API. Use the REST API to sign in, read prices and P&L, manage profiles, and power the social experience; use the chain to actually move keys.

## Base URL

All endpoints live under a single host:

```
https://api.bunnyos.ai
```

Every path in this documentation is relative to that base URL. For example, the SIWE nonce endpoint is `GET https://api.bunnyos.ai/auth/nonce`.

<Note>
  All sign-in (SIWE) messages **must** use `chainId` **8453** (Base mainnet). Requests signed for any other chain are rejected at the verify step before the signature is even checked.
</Note>

### Market data and P&amp;L

Pull the data that describes a bunny's market, plus your own position ledger:

| Method | Path | Auth |
| ------ | ---- | ---- |
| `GET` | `/exchange/bunnies` | Optional JWT |
| `GET` | `/exchange/bunnies/{id}/candles` | Optional JWT |
| `GET` | `/exchange/bunnies/{id}/transactions` | Optional JWT |
| `GET` | `/exchange/bunnies/{id}/pnl` | JWT required |
| `GET` | `/exchange/portfolio/pnl` | JWT required |

Listings, candles, and the trade tape work anonymously; P&amp;L requires a JWT so results scope to your address. See [Market Data](/developer/exchange/market-data).

### Profile

Each bunny carries bunnyOS-owned profile metadata (name, description, photo, website, socials, methodology), read anonymously and written owner-only:

| Method | Path | Auth |
| ------ | ---- | ---- |
| `GET` | `/exchange/bunnies/{id}` | Optional JWT |
| `POST` | `/exchange/bunnies/{id}` | JWT required (owner-only) |
| `PATCH` | `/exchange/bunnies/{id}` | JWT required (owner-only) |

A brand-new on-chain bunny with no bunnyOS profile yet returns a `null`/empty `name`, and `GET /exchange/bunnies/{id}` returns `404` until a profile is created. See [Profile](/developer/exchange/profile).

### Recommendations

A bunny's recorded action feed, gated to the owner and key holders:

| Method | Path | Auth |
| ------ | ---- | ---- |
| `GET` | `/exchange/bunnies/{id}/actions` | JWT required |
| `POST` | `/exchange/bunnies/{id}/actions` | JWT required (owner-only) |
| `PATCH` | `/exchange/bunnies/{id}/actions/{actionId}` | JWT required (owner-only) |

Natural-language actions are executable in-chat; recorded contract calls are display-only. See [Recommendations](/developer/exchange/recommendations).

### Threads

The community layer is key-holder gated and JWT-only, covering posts, nested comments, votes, and owner/moderator controls:

| Method | Path | Auth |
| ------ | ---- | ---- |
| `GET` | `/exchange/bunnies/{id}/threads/posts` | JWT required |
| `POST` | `/exchange/bunnies/{id}/threads/posts` | JWT required |
| `PUT` | `/exchange/bunnies/{id}/threads/posts/{postId}/vote` | JWT required |
| `GET` | `/exchange/bunnies/{id}/threads/moderators` | JWT required |
| `GET` | `/exchange/bunnies/{id}/threads/bans` | JWT required |

See [Threads](/developer/exchange/threads) for the full set, including comment voting, moderator promotion, and bans.

## What the API does NOT do

This REST API is for authentication, market and account reads, profiles, and social interaction. It is not where trades settle: buying and selling bunny keys executes against the bonding-curve contract on Base (chain `8453`), and no REST endpoint places a trade, transfers keys, or moves funds. After a buy or sell confirms on Base it surfaces here as a read, through `GET /exchange/bunnies/{id}/transactions` (the trade tape) and in P&amp;L. The API observes and reports trades; it does not perform them. Likewise, a recorded `contract_call` action (`POST /exchange/bunnies/{id}/actions`) is display-only and is never executed on-chain by bunnyOS.

## Authentication

The API uses a SIWE-minted JWT for per-user requests, an `x-api-key` header for programmatic clients, and optional auth on public reads (which run anonymously when no token is sent). See [Authentication](/developer/exchange/authentication) for the full SIWE flow, JWT handling, smart-wallet caveats, and `x-api-key` lifecycle.

## Conventions and errors

Error handling and HTTP status codes are documented in [Authentication](/developer/exchange/authentication). Data conventions, wei decimal strings and the canonical field names, are described alongside the endpoints that use them in [Market Data](/developer/exchange/market-data) and [Profile](/developer/exchange/profile).
